# 1 Architecture As A Code

### Вариант 6 — Сервис доставки

## Описание варианта

Система должна предоставлять API для выполнения следующих операций:

- создание нового пользователя;
- поиск пользователя по логину;
- поиск пользователя по маске имени и фамилии;
- создание посылки;
- получение посылок пользователя;
- создание доставки от пользователя к пользователю;
- получение информации о доставке по получателю;
- получение информации о доставке по отправителю.

---

## Цель работы

Целью работы является описание архитектуры программной системы в стиле **Architecture as Code** с использованием **Structurizr DSL**, а также проектирование системы сверху вниз: от контекста системы до контейнеров и их взаимодействия.

---

## Общая идея системы

Разрабатываемая система представляет собой веб-сервис доставки, в котором пользователь через веб-интерфейс взаимодействует с внутренними сервисами системы.  
Архитектура построена по контейнерному принципу с разделением ответственности между несколькими backend-сервисами.

Основной сценарий работы системы связан с созданием доставки. Для оформления доставки необходимо:

1. проверить существование отправителя и получателя;
2. создать или получить информацию о посылке;
3. выполнить проверку/подтверждение оплаты через внешнюю платежную систему;
4. сохранить информацию о доставке;
5. при необходимости отправить уведомление пользователю.

Такой подход позволяет разделить бизнес-логику по отдельным контейнерам и сделать систему более масштабируемой и сопровождаемой.

---
### Customer
Пользователь сервиса доставки, который работает с системой через веб-интерфейс.  
Он может:

- зарегистрироваться в системе;
- искать других пользователей;
- создавать посылки;
- оформлять доставки;
- просматривать информацию о своих доставках.

---

## Внешние системы

Система взаимодействует с двумя внешними системами:

### Payment System
Внешняя платежная система, которая используется для подтверждения или обработки оплаты доставки.

### Email
Внешняя система уведомлений, которая используется для отправки пользователю сообщений о статусе заказа или доставки.

---

## Описание программной системы

### Delivery Service
Основная программная система, обеспечивающая создание и обработку посылок и доставок между пользователями.  
Система предоставляет единый пользовательский интерфейс и набор внутренних сервисов для управления предметной областью.

---

## Контейнеры системы

### Web Application (Frontend)
Веб-приложение, через которое пользователь взаимодействует с сервисом доставки.  
Отвечает за отображение интерфейса, отправку запросов на backend и получение результатов.  
Предполагаемые технологии: **React.js / Vue.js**.

### User Service
Сервис управления пользователями.  
Отвечает за:

- создание пользователей;
- поиск пользователя по логину;
- поиск пользователя по имени и фамилии.

Предполагаемые технологии: **C++ / Userver**.

### Parcel Service
Сервис управления посылками.  
Отвечает за:

- создание посылки;
- получение списка посылок пользователя.

Предполагаемые технологии: **C++ / Userver**.

### Delivery Service (Backend)
Основной backend-сервис бизнес-логики доставки.  
Отвечает за:

- создание доставки;
- поиск информации о доставке по отправителю;
- поиск информации о доставке по получателю;
- координацию взаимодействия между внутренними сервисами;
- обращение к внешней платежной системе;
- отправку уведомлений.

Предполагаемые технологии: **C++ / Userver**.

### Redis Cache
Кэш, используемый для ускорения доступа к данным по доставкам и пользовательским запросам.  
Позволяет снизить нагрузку на backend и базу данных при повторяющихся обращениях.

### User Database
База данных пользователей.  
Используется сервисом `User Service` для хранения и получения информации о пользователях.  
Технология: **PostgreSQL**.

### Parcel Database
База данных посылок.  
Используется сервисом `Parcel Service` для хранения информации о посылках.  
Технология: **PostgreSQL**.

### Delivery Database
База данных доставок.  
Используется backend-сервисом доставки для хранения информации о созданных доставках и связях между отправителями, получателями и посылками.  
Технология: **PostgreSQL**.

---

## Технологии и протоколы

Для контейнеров и связей в архитектуре выбраны следующие технологии:

- **Frontend** — React.js / Vue.js
- **Backend-сервисы** — C++ / Userver
- **Базы данных** — PostgreSQL
- **Кэш** — Redis

Используемые протоколы взаимодействия:

- между пользователем и веб-приложением — **HTTPS**;
- между frontend и backend — **HTTPS/REST**;
- между backend-сервисами — **gRPC/HTTP**;
- между сервисами и базами данных — **JDBC/SQL**;
- между backend и Redis — **RESP / Redis protocol**;
- между системой доставки и внешней платежной системой — **HTTPS/REST**.

## Основные сценарии использования

Система должна поддерживать следующие основные пользовательские сценарии:

### 1. Регистрация нового пользователя
Пользователь вводит данные для регистрации, после чего запрос передается в `User Service`, где создается новая запись в базе пользователей.

### 2. Поиск пользователя
Пользователь может найти другого пользователя по логину или по маске имени и фамилии. Для этого frontend обращается к `User Service`.

### 3. Создание посылки
Пользователь вводит параметры посылки, после чего информация сохраняется через `Parcel Service`.

### 4. Создание доставки
Пользователь инициирует создание доставки, указывая отправителя, получателя и посылку.  
Система проверяет корректность данных, взаимодействует с платежной системой и сохраняет информацию о доставке.

### 5. Получение информации о доставках
Пользователь может получить список или сведения о доставках по отправителю или получателю.

## Архитектурно значимый сценарий

В качестве архитектурно значимого был выбран сценарий **создания доставки от пользователя к пользователю**.

Последовательность взаимодействия контейнеров в этом сценарии выглядит следующим образом:

1. `Web Application` отправляет запрос на создание доставки в backend-сервис.
2. `Delivery Service` обращается в `User Service` для проверки существования отправителя и получателя.
3. `User Service` запрашивает данные из `User Database` и возвращает результат проверки.
4. `Delivery Service` обращается в `Parcel Service` для создания записи о посылке.
5. `Parcel Service` сохраняет данные в `Parcel Database` и возвращает идентификатор посылки.
6. `Delivery Service` обращается во внешнюю `Payment System` для подтверждения оплаты.
7. После успешной оплаты `Delivery Service` сохраняет информацию о доставке в `Delivery Database`.
8. Затем система может инициировать отправку уведомления пользователю через `Email`.

Данный сценарий является важным, так как затрагивает основную бизнес-логику системы и демонстрирует взаимодействие сразу нескольких контейнеров и внешних систем.

## Обоснование архитектурных решений

Данная архитектура является *** сервис-ориентированная архитектура с признаками микросервисной ***

Выбранная архитектура основана на декомпозиции системы на отдельные контейнеры по функциональной ответственности.

Такой подход дает следующие преимущества:

- упрощает сопровождение системы;
- позволяет независимо развивать сервисы пользователей, посылок и доставок;
- облегчает масштабирование наиболее нагруженных частей;
- делает интеграцию с внешними системами более явной;
- позволяет кэшировать часто запрашиваемые данные.

Использование отдельных баз данных для разных сервисов уменьшает связанность между контейнерами и делает архитектуру ближе к микросервисному подходу.

Redis используется как дополнительный уровень оптимизации производительности.  
Внешняя платежная система и email-сервис вынесены за пределы системы, так как они являются внешними зависимостями.
